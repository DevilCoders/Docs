Gate (или Zora Gate) - единое место входа запросов от пользователей в Зору.

Запрос приходит через один из множества входных интерфейсов (input) и дальше улетает в соответствующий сервис через один из нескольких выходных интерфейсов (output). По дороге запрос взаимодействует с некоторыми кубиками (авторизация, dns resolve и т.п). Потом через единый интерфейс возвращается ответ из сервиса и доставляется до нужного входного интерфейса.

Кубики общаются между собой через акторные эвенты. Для каждого типа кубиков есть интерфейс (набор событий), который должны поддерживать все кубики данного типа. На другие особенности реализации нельзя завязываться. То есть внутри одного типа кубики взаимозаменяемы (может ломаться бизнес-логика, но не должен ломаться пайплайн). Каждый тип кубиков описывается в своей директории. Интерфейс для типа кубиков задаётся в interface.h этой директории.

Как следствие входящий интерфейс должен уметь преобразовать запрос от пользователя в TRequest и в обратную сторону TResponse в ответ. И, в общем-то, это всё, что он должен уметь.

Пайплайн можно понять почитав код request'а.

Картинка есть тут: https://wiki.yandex-team.ru/zora/techdoc/2021/arxitektura/#zora

Кубики:
* auth
* quota
* input
* output
* job
* job_generator - создаёт разные типы джобов в зависимости от запроса

Прочее:
* config - основной конфиг приложения. Доступен в любом месте через TGateConfig::Get(). 
* counters - счётчики специфичных для гейта компонент. В частности, поисточниковые счётчики. Доступны всюду.
* proto - специфичные для гейта протобуфы
* server - там main
