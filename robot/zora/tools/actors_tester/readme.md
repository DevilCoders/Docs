Тулзовина для тестирования акторсистемы (NZora::ActorSystem).
Есть три сценария работы: pinger, requester, bottleneck. Все три могут работать одновременно, включаются и настраиваются флагами запуска.

Над этим всем есть счётчики. Они сделаны несколько сложным образом - каждый актор собирает счётчики локально и раз в 5 секунд посылает в специальный актор TMonitoringActor. Сделано это потому что реальной работы внутри обработки сообщений нет, поэтому посчёт метрик на каждый запрос может быть дороже обработки запроса. В общем случае в проде так делать не стоит.

1. Pinger. (pinger.h/cpp)

    Создаются NPingers пингеров. Каждый пингер знает адрес 10 случайных других пингеров. При получении пинга пингер выбирает очередного адресата и шедулит ему доставку сообщения через Delay. Если Delay == 0, то вызывается обычный Send.

    Изначально в систему посылается NPings пингов. После этого они начинают случайным образом ходить по пингерам через вышеописанную схему.

    Для пингов считаем задержку - разницу между прогноззируемым и реальным временами доставки.

    Схема с Delay=0 мы по максимуму нагружаем акторсистему.

    Схема с Delay=0.5s, NPings=10k порождает поток из 20k rps сообщений.


2. Requester (request.h/cpp)
    
    Есть запросы. Каждый запрос создаётся, раз в PingInterval посылает себе сообщение ("меняет статус"), через Timeout умирает. Перед смертью отправляет метрики. Умолчания настроены так, чтобы каждый запрос отправлял себе 10 пингов перед смертью.

    Есть TRequestCreator. Он просыпается раз в секунду и создаёт пачку из NRequestsRate.

    Изначально создаются два TRequestCreator. Slow - это долгоживущие запросы (120s), Fast - быстрые (3s).

3. Proxy (proxy.h/cpp)
    Имитирует BottleNeck, то есть синглтоновых акторов с высокой нагрузкой.
    
    Есть TProxy, в него можно послать TPing, он подождёт WaitTime и пошлёт обратно TPong.
    
    Создаётся NSenders штук TSender. Каждый из них просыпается раз в секунду и посылает Rate запросов в TProxy.
