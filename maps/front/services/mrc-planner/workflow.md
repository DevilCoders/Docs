# Рабочий процесс

## Регламент

1. Все фичи делаются в отдельных ветках и затем вливаются в dev.
2. Помните, что все коммиты из dev обязательно попадают в следующий релиз!

## Процесс работы над задачей

Когда задача берется в разработку, для нее создается отдельная ветка в репозитории. Название ветки обязательно должно начинаться с логина разработчика, далее, через точку, указывается код задачи в стартреке. Также название полностью должно быть в нижнем регистре для корректной работы стендов.

пример:

```
ivanovivan.mapsmrc-1234
```

Все ключевые договоренности, которые были достигнуты во время работы над задачей с другими командами, стоит фиксировать в стартреке, для того чтобы облегчить работу QA-инженерам и разработчикам, которые не принимали непосредственное участие в решении задачи.

Перед вливанием ветки в dev обязательно прохождение ревью (в рамках карт разработан [набор рекомендаций](https://wiki.yandex-team.ru/maps/dev/ui/workflow/review/#codereview) для его проведения). Для проведения ревью создаются pull request в Аркануме.

В качестве названия PR лучше указывать полное название задачи, решаемой в рамках данного PR (можно скопировать из стартрека). В идеале 1 задача в трекере = 1 PR. Если изменений много и все они маленькие (буквально несколько строчек), их можно объединить в единый PR, главное дать ему понятное название. Решаемые задачи можно указать в виде ссылок на задачи в трекере в описании пулл-реквеста.

После создания PR в [Sandbox](https://sandbox.yandex-team.ru/tasks/?order=-updated&created=7_days&type=SANDBOX_CI_DOCKER_QLOUD_DEPLOY&page=1&pageCapacity=20&forPage=tasks&owner=MAPS_FRONT) создаются задачи, которые запустят линтеры и сборку.

Если какие-то из проверок сломались из-за внесенных в проект измененний, необходимо их починить.
