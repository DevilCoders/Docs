Обёртка над `YtClient` со следующими фичами
1. Упрощённый способ передачи локальных файлов/директорий и заказа файлов кипариса.
2. Запуск операций, использующих карточный граф.
3. Костыли

Упрощённый способ передачи локальных файлов/директорий и заказа файлов кипариса.
===

В качестве mapper'а и reducer'а можно использовать классы `BinaryCmd` и `PythonCmd` для бинарников и питоновских классов-операций соответственно.
В качестве аргументов помимо обычных значений им можно передавать `LocalFile`, `LocalDir`, `YtFile`, `YtNodeFile` и `YtNodeDir`, которые будут заменены на "правильные" значения, доступные при запуске джобы. Это приведёт к отправке соответствующих файлов/директорий, заказу файлов кипариса, а также выставлению правильного `tmpfs_size` и копрованию локальных файлов ноды при указании использовать `tmpfs`.<br>
Также каждая операция понимает аргумент `title` - заголовок операции (просто более удобный способ выставить `title` в спеке)

Пример запуска бинарника:
```
ytc.run_map(
    BinaryCmd(
        './mybinary',
        params={
            '--foo': LocalFile('/home/me/local'),
            '--bar': LocalDir('/home/me/dir'),
            '--cypress': YtFile('//home/me/cypress_file'),
        },
        environment={
            'SOME_ENV_VAR': LocalFile('/home/me/some_file'),
        },
        attach=[LocalFile('/home/me/some-file-to-place-near-binary)],
    ),
    '//src', '//dst',
    title='Custom operation title',  # То же, что `spec={'title': 'Custom operation title'}`
    description={
        'help': 'custom description of operation',
        'binary': 'mybinary',
        'url': utils.yson_url('https://yandex.ru'),  # Кликабельная ссылка в веб-интерфейсе операции YT
    },  # То же, что и `spec={'description': {...}}`; если в `default_spec` уже есть описание, оно будет смержено
)
```

Добавит в `local_files` запускаемой операции бинарник и его зависимости, локальные файлы и запакованную (tar) директорию, закажет в `yt_files` кипарисный файл.<br>
Перед запуском директория будет распакована, а бинарник запустится с такими аргументами:
```
SOME_ENV_VAR=./some_file ./mybinary --foo ./local --bar ./dir --cypress ./cypress_file
```

Пример запуска питоновского класса-операции:
```
class MyOp(object):
    def __init__(self, file, dir=None):
        self.file = file
        self.dir = dir

    def start(self):
        pass  # prepare file/dir

    def call(self, value):
        pass  # use

ytc.run_map(
    PythonCmd(
        MyOp,
        LocalFile('/home/me/local_file'),
        dir=LocalDir('/home/me/local_dir'),
    ),
    '//src', '//dst',
)
```

В качестве аргументов `file` и `dir` в конструктор придут `./local_file` и `./local_dir`.<br>
Учитывайте, что конструирование происходит до запуска операции локально, т.е. инициализацию, использующую заказанные файлы/директории, надо проводить в `start`.

Запуск операций, использующих карточный граф.
===
В модуле `graph` есть класс `GraphCmd` для запуска питоновских операций, использующих граф. Отнаследуйтесь от него, инициализируйте, передав нужные файлы графа (по умолчанию `DATA` и `TOPOLOGY`), и можно будет использовать соотвествующие свойства `GraphCmd`: `graph`, `graph_data`, `persistent_index`, ...

Костыли
===
* Сбрасывается `expiration_time` при копировании временной таблицы
* `run_reduce` для питоновских операций работает с пустым ключом (все записи придут в одну джобу)
* Автоматически удаляются временные таблицы по выходе из контекста
* `run_sort` умеет не сортировать уже посортированную по тем же ключам таблицу, для этого надо указать флаг `always_sort=False`. Вместо этого таблица будет скопирована. Поведение по умолчанию можно регулировать, передав при создании YtContext аргумент `always_sort=...`. Если выставить его в `False`, то все сортировки не будут сортировать уже сортированные по тем же ключам таблицы.
* `copy` кидает исключение, если путь до исходной таблицы содержит селекторы колонок / строк, так как копирование в этом случае делает не то, что ожидается: копирует таблицу целиком. Как правило это не то, чего хочет пользователь функции. Добавлен флаг `use_merge=True`, который в этом случае сделает `run_merge` исходной таблицы в целевую, т.о. "применив" фильтры по колонкам и строкам.
* `YtContext` принимает функцию `prepare_spec`, позволяющую пропатчить спеку. Функция на вход принимает:
    - `self`
    - общая спека
    - спека операции (маппер/редьюсер)
    - `PreparedCmd`, если был использован `BinaryCmd`/`PythonCmd`, иначе `None`
    Возвращать ничего не должна, может модифицировать спеки
