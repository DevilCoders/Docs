## Правила оформления задач:
{% note warning %}

### Статус задачи всегда должен быть актуальным

{% endnote %}
1. У всех задач в работе, нужно указать текущий спринт в поле `Agile - Спринт`.
2. У всех задач со статусом `Открыт` и далее должны быть указаны `Agile - Story Points`
3. Для продуктовых задач необходимо указать менеджера.
2. Заголовок пулл-реквеста должен начинаться с номера задачи в [Стартреке](https://st.yandex-team.ru).
   [Пример](https://st.yandex-team.ru/HOME-75303):
   `HOME-75303: [ desktop ] Поддержать новый профиль с Яндекс ID`
3. Если разработчик самостоятельно тестирует задачу, он сам выставляет статусы `Тестируется` - `Готово к коммиту` и статусы `Testing at RC` - `Готово к выкладке`
4. При написании функциональных тестов необходимо в описании задачи перечислить тесткейсы для проставления тестированием `Test coverage ratio` / `Доля покрытия автотестами`. Если вы написали тесты, убедитесь, что тестирование его проставило. Технические и другие задачи, в которых мы пока не пишем тесты, помечаются тегом `#notest`
5. Перед передачей в тестирование укажите план тестирования: бета, флаги включения фичи, на что обратить внимание. [Пример](https://st.yandex-team.ru/HOME-72502)
6. При необходимости укажите план релиза. [Пример](https://st.yandex-team.ru/HOME-77808)
7. Если для задачи необходимы изменения в МАДМ, укажите это в поле `Тестирование-Комментарий разработки`. [Пример](https://st.yandex-team.ru/HOME-77820): `ПОСЛЕ выкладки включить опцию disable_topnews_subgraph_http_response`
8. Perl, Go и Apphost (графы) выкладываются отдельными релизами. поэтому под каждый компонент нужно создавать отдельную задачу.
   Например: в задаче требуеются правки в Perl и в Go. Нужно клонировать основной тикет, добавить пометку `[go]` в названии и поставить оценку `0 sp`. В итоге получится два тикета: один с правками в Perl, второй с правками в Go.


## Статусы задач
Каждая задача в Стартреке проходит через статусы:

`
Новый - Открыт - В работе - Ждет ревью - Ревью проведено - Можно тестировать - Тестируется - Готово к коммиту - Commited - Release to RC - Testing at RC - Готово к выкладке - Выложили - Закрыт
`

Также есть статус `Треубется информация`, который выставляется, если работа над задачей не может быть продолжена без получения дополнительной информации.

Подробнее про статусы: https://wiki.yandex-team.ru/Morda/workflow/

## Особенности
### Perl:
Код живет в [Гитхабе](https://github.yandex-team.ru/morda/main).

Продуктовые задачи должны быть отмечены тегом `#test4product`

Разработчик выставляет статусы:
`В работе` и `Можно тестировать`. Остальные статусы проставляеются автоматически или тестировщиком.


### Go:
Код живет в [Аркадии](https://a.yandex-team.ru/arc_vcs/portal).

Продуктовые задачи должны быть отмечены тегом `morda-go`

Разработчик выставляет статусы:
`В работе`, `Можно тестировать`, `Commited`, `Закрыт`.

Если требуется тестирование на RC, сообщите об этом дежурному платформы после коммита.
Узнать о том, что таск на RC, можно по отбивке в чате ["Покатушки Морды"](https://t.me/joinchat/BlyRek7XrnZIJpDaD8C3kQ).


### Apphost (графы):
Код живет в [Аркадии](https://a.yandex-team.ru/arc_vcs/apphost/conf/verticals/MORDA).

Разработчик выставляет статусы:
`В работе`, `Можно тестировать`, `Commited`, `Закрыт`.

Когда таск будет готов к коммиту, предупредите дужрного Платформы о том, что необходим релиз графа.
Если требуется тестирование на RC, сообщите об этом дежурному платформы **до коммита**.
После разрешения дежурного отправляйте коммит на мердж в транк.
Призовите тестировщика, когда таск выедет на RC.
