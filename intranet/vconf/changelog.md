1.4.6
-----
 * VCONF-847: Взятие лока при создании звонка с event_id

Данный pr пытается решить проблему с возможностью наличия более одного звонка для одной встречи.

При наличии 2х конкуретных запросов первый запрос берёт лок и входит в защищённую секцию кода, в этой секции создаётся звонок. Второй запрос блокируется в ожидании выполнения защищённой секции кода. 

С точки зрения логики приложения это выглядит следующий образом:
Если первый запрос создал звонок успешно, то второй запрос отдаёт на фронт id существующего звонка(и фронт редиректит в нужный звонок), иначе второй запрос сам пытается создать звонок.

При данном подходе потенциально могут возникнуть две проблемы при которых нельзя будет собрать звонок для встречи из-за висящего лока.
1. Инстанс умрёт во время выполнения защищённой секции кода
2. Что-нибудь зависнет во время выполнения защищённой секции кода (не знаю на сколько это возможно)

Поскольку невозможность создания звонка - большее зло, чем конкуретные звонки, хочется предусмотреть эти проблемы.
Первую проблему решает параметр timeout. Он задаёт время жизни транзакции в yt с момента последнего пинга.
Вторую проблему решает block_timeout. Если запрос будет ждать лока дольше block_timeout, то защищённая секция выполнится так, будто никакого лока тут и нету, то есть возвращаемся к схеме с возможными конкурентными звонками. 
 


Для реализации этой фичи потребовался небольшой рефакторинг:
\- код для проверки существующего звонка для той же встречи был вынесен из функции _fetch_event_external_id в _check_another_call_exists
\- процесс создания звонка был вынесен в вспомогательную функцию _create, вызов которой происходит вместе с взятием лока, если данный звонок собирается для встречи  [ https://a.yandex-team.ru/arc/commit/7108395 ]
