# Глоссарий
* `Pull схема` - актуальная схема получения описания оферов от партнеров Маркета. Робот периодически по собственному расписанию скачивает прайс-листы и размещает оферы из них на выдаче. Партнер может управлять периодом скачивания. Если робот не может скачать фид, то спустя некоторое время оферы пропадают с выдачи.
* `Push схема` - схема обновления данных в маркетном хранилище оферов (индексаторном). Партнер передает прайс-лист с изменившимися оферами в необходимые ему моменты. Подразумевает наличие состояния с оферами на стороне Маркета и является инструкцией по его изменению.
* `Complete Feed` - полный список оферов к размещению. Оферы из прайс-листа заменяют предыдущие оферы на выдаче, ненайденные в прайс-листе оферы удаляются. Может использоваться в пул и пуш-схеме.
* `Update Feed` - фид с частичным списком оферов. Изменяются только оферы, указанные в фиде. Оферы могут быть добавлены, заменены, удалены. Может использоваться только в пуш-схеме. Применяется к текущему размещаемому списку оферов партнера. По формату практически не отличается от обычного фида, за исключением магазинных категорий. Complete Feed является частным случаем Update Feed с дополнительной инструкцией по предварительному удалению всех оферов.
* `Patch Feed` - фид с частичным содержанием оферов. Частный случай Update Feed, рассматриваемый отдельно из-за необходимости дополнительной разработки. Прайс лист может содержать только часть офера, которую хочет обновить партнер. Например: цена или тайтл. Сопоставление офера происходит по его идентификатору.
* `DataCamp` - "Campaign Data". Имя собственное, выбранное для хранилища оферов партнеров в индексаторе для публикации на выдаче. Необходимо для отличия хранилища от других хранилищ оферов в Маркете, а также для удобства разработчиков.
* `DYT` - Dynamic Yandex Tables. Способ хранения оферов в хранилище индексатора - динамические сортированные реплицированные таблицы YT.
* `PQ` - Persistent Queue. Отказоустойчивый высокопроизводительный транспорт доставки данных между сервисами. В Яндексе используется собственная реализация Logbroker. Для проекта логброкер выбран для доставки оферов в хранилище асинхронным способом.
* `Перематчинг` - условный термин для обновления привязки офера к маркетному контенту. Индексатор получает из УльтраКонтроллера соответствие офера маркетным карточкам. Перематчинг надо производить при изменении офера или раз в некоторый период (около суток). Необходимость регулярного перематчинга обусловлена изменением маркетного контента и переобучением роботов IR.
* `Переобогащение` - условный термин для обновления всей маркетной информации по оферу, необходимой для индексации. Включает перематчинг, обновление опций доставки и применение другой бизнес-логики
* `Консистентность` - это состояние оффера в ЕОХ, когда он считается полностью обогащенным данными для обработки конпонентом-потребителем. [Консистентность в формате Datacamp-оффера](https://a.yandex-team.ru/arc/trunk/arcadia/market/idx/datacamp/proto/offer/OfferStatus.proto?rev=r7724646#L188).
