Новая схема с definitions.ts.

Для обновления используйте `../definitions-update/update-definitions.sh` и не забудьте закоммитеть.

Как было до этого: definitions.ts генерились из кода бэкенда. Из проблем - это долго и это сложно интегрировать в сборку фронта.

Как сейчас:
- definitions.ts хранятся прямо в Аркадии в отдельном модуле `mbo-react-ui/src/definitions`
- фронт просто об них собирается
- в бэке будет тест, который сверяет definitions.ts с тем, что в скомплированном коде и падает, если начало расходиться
- можно запустить main() в этом же тесте, чтобы обновить `definitions.ts` (ему нужно окружение ARCADIA, полезно прямо в ~/.profile|~/.zshrc указать)
  - либо запустить `../definitions-update/update-definitions.sh` - ему не нужно окружения.

Плюсы:
- оторвём сборку фронта из ya make-а целиком в newCI
    - не будет собираться реакт в обычной сборке (быстрее выкладка в МТ, тестинг)
- будет история изменений по definitions.ts
- мы всё ещё сохраняем контроль на уровне системы сборки, что изменения бэка не ломают фронт
    - если поменялся фронт - он должен быть совместим с definitions.ts
    - если поменялся бэк - то надо обновить definitions.ts, что приведёт к сборке фронта с ними

Минусы:
- обновлять пока руками (запуская main() в тесте)
- иногда может увеличивать цикл разработки, если забыли и упал тест про это - нужно пофиксить и допушить (как забытый чекстайл)

Развитие:
- м.б. получится автообновлять через бота (но это не просто)
- как стратегическая история, в варианте разных команд фронта и бэка - я бы смотрел в OpenAPI
