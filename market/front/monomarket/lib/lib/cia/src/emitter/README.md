Мидлварь для работы с "новой-новой" (унифицированной) метрикой
==

Расшифровка идентификатора метрики
===
```
    someZoneA:someZoneB~some:ACTION
        |    |  |      |  |     |
        |    2  |      3  |     5
         -------          4
            |
            1

    1 имена зон, camelCase
    2 разделитель зон
    3 разделитель частей цели <zones>~<action>
    4 префикс экшена (за исключением сервисного (@<service>/, i.e. @mobile/GetSome/ACTION -> getSome:ACTION), camelCase
    5 имя экшена в UPPER_CASE (приватные экшены, начинающиеся с #, запрещены к отправке)
```

Описание
===
Взято из https://st.yandex-team.ru/MARKETFRONTECH-489

Решили, что цель будем формировать как  someZoneA:someZoneB~some:ACTION , то есть:
- Наименование зон и префикса в  camelCase .
- Наименование экшена в  UPPER_CASE
- Зоны разделяются  : .
- Части (зоны и экшен) разделяюся   ~  .
- (при наличии) Из экшена убираем указание репозитория (  @cia/ ,  @mobile/ ), префикс "some" оставляем, чтобы избежать коллизии
- Не шлём приватные экшены - виджетные экшены, начинающиеся с  # . Если экшен становится бизнес-событием, то его необходимо делать глобальным.
- Параметры шлются сгруппированные по зонам (как в белом десктопе), а не мёржатся (как в белом таче).
Имя цели:  someZoneA:someZoneB~some/ACTION
```json
{
  someZoneA: {someZoneAParams},
  someZoneB: {someZoneBParams}
}
```

В случае дублирования названия зон, параметры отправлять массивом объектов параметров для каждой зоны в порядке их появления от верхнеуровневой зоны

Имя цели:  someZoneA:someZoneB:someZoneA~some:ACTION
```json
{
  someZoneA: [{someZoneAParams - параметры первой зоны someZoneA}, {someZoneAParams - параметры второй зоны someZoneB}],
  someZoneB: {someZoneBParams}
}
```

- Если у экшена есть payload, кладем его в метрику с ключем actionPayload

Яндекс.Метрика вводит ограничение на использование некоторых символов в названии цели  `/[/&=?#]/` . Поэтому в качестве разделителя выбран  `:`
