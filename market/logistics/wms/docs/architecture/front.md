## Слои
Слои перечислены изнутри наружу. Внешние слои могут (и по идее должны) зависеть от внутренних слоёв. Внутренние слои не должны зависеть от внешних.
### **Entity**
Для каждой сущности внутри каждого проекта создается папочка со слеюущими файликами (всё кроме **index** - опционально)
   * **types** - описание фронтовых типов
   * **atoms** - описание хранилищ для фронтовых данных
   * **actions** - описание действий для работы с хранилищами
   * **getters** - функция для доставания/консолидации/проверки условий в данных
   * **index** - экспортит всё
### **Api**
Для каждого из контроллеров например /packing создается папочка. Внутри каждой папочки так же могут создаваться папочки
   * **types** - описание бэковых типов
   * **mock** - моки фронта
   * **schema** - мапперы для перевода бэковых типов во фронтовые и обратно (лучше делать на уровне апи, т.к. не хотелось бы, чтобы слой сущностей зависел от типов бэка)
   * **index** на верхнем уровне - объект апи с методами для контроллера
### **fetchActions** - завязанные на стор обёртки над вызовами апи
   * подумать, мб стоит засунуть в апи и название получше придумать
### **useCases** - непосредственные действия пользователя (бизнес логика)
   * Этот уровень можно пропускать, если есть только действия со страницей
### **pages** - уровень страницы
   * Может содержать свои атомы и экшены относящиеся конкретно к этой странице. Так же может содержать в себе useCases. Так же может содержать в себе папочки с контейнерами и компонентами для декомпозиции сложных страниц.

==Папочки
Большая часть названий папочек совпадает со слоями, но есть frontend specific
## **components**
   * непосредственно компоненты с отображением и логикой с ним связанной, в них **нельзя использовать сторы** (управляется через пропсы)
### **containers**
   * Контейнеры используются для создания изолированного модуля с логикой, **завязанной на стор**. В качестве представления используют компоненты из папочки **components**
### **views**
   * Цельное отображение страницы, не содержащее никакой логики
### **routes**
   * Место с маппингом урл -> компонент + ленивое подключение атомов
### **fsm**
   * Объект с функциями переходов между страницами
### **reducer**
   * Обычно объединение всех используемых в проекте атомов
   * (подумать, мб стоит перенести useCases сюда)
### **utils**
   * (подумать, мб имеет смысл объединить все utils, а то что не utils - перенести)
