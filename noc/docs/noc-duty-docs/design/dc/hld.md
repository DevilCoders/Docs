# 1. Архитектура ДЦ

## 1.1. Правила ведения дизайна

1.1.1. Дизайн должен содержать четко описанные правила и рекомендации

1.1.2. Каждое правило или рекомендация должна иметь свой номер

1.1.3. Формат правила или рекомендации должен содержать: первое предложение - четко сформулированное требование, пустую строку, дополнение или пояснение к требованию.

```
+---------------------------------+
| Обязательное поле               |
|       Правило/Рекомендация      |
|                                 |
+---------------------------------+

+---------------------------------+
| Дополнительное поле             |
|       Пояснения/Дополнения      |
|                                 |
+---------------------------------+
```

1.1.4. Необходимо поддерживать сквозную нумерацию правил и рекомендаций

1.1.5. Не допускается дублирование номеров правил

1.1.6. В конце номера правила обязательно ставится точка `.`

1.1.7. Разделителем правила является символ точки `.`

## 1.2. Общие принципы

1.2.1. Необходимо обеспечивать заданные SLA.

1.2.2. На оборудовании, маршрутизирующем трафик внутри ДЦ, используется только IPv6.

1.2.3. L3 до стоечного коммутатора. Граница маршрутизируемого домена перенесена на стоечные коммутаторы, которые выполняют роль "first hop router"

1.2.4. Только BGP. Единственным протоколом маршрутизации внутри ДЦ является BGP

1.2.5. Внутри ДС не используется L2.

1.2.6. Сегментация проектов производится на уровне IP-адресации и на уровне HBF

## 1.3. Термины и определения

1.3.1. Датацентр (ДЦ, DC) - совокупность инженерных, серверных и сетевых систем. Может состоять из одного здания, а может и из нескольких.

1.3.2. Очередь - здание внутри ДЦ.

1.3.3. Модуль - часть здания внутри очереди. Независимо функционирующая единица вводимой в эксплуатацию части ДЦ. ДЦ расширяются модулями, по 130-160 стоек
