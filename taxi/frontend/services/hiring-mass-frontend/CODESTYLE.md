## Экпорт

**При экспорте переменных/методов/etc давать им осмысленные названия, содержащие контекст**

**Не делаем default export там, где без него что-то не работает**

Примеры экпортируемых элементов кода **bad => good**

## Реэкспорт

```
/static
  /project/search
    /page.js
  /shared/ui
    /button.js
    /panel1
      /hook.js
      /panel1.js
      /index.js
    /panel2
      /hook.js
      /panel2.js
      /index.js
```

index.js файлы импортируют из текущей папки и реэкспортируют только то что является публичным интерфейсом
если весь публичный интерфейс помещается в одном файле (например button.js) то папка и index.js не нужен
все либы (например если button это amber-blocks/button) импортятся в shared/ui/button и экспортятся, а конечные потребители импортят именно из shared

## Мемоизация

Не используем везде **memo**, **useCallBack** и **useMemo**, а используем только когда есть основания:

-   При наличии оснований в измерениях производительности
-   Когда есть внутренний стейт, обновляющий часть компонента
-   Когда это нужно для правильной работы эффектов
-   Когда в функции есть внутренний стейт (дебаунс, троттлинг)
-   Когда очевидно это просто нужно
-   useCallback - например передача функции глубоко в дерево
-   useMemo при обработке больших данных
-   memo при рендере сложного компонента

## Effector

-   Не используем forward, а используем sample
-   Events называем в прошедшем времени. Пример: **filterRestored**

## Переводы

1.  Для каждой страницы свой файл. Пример: search-page (в соответствии с названием бандла)
2.  Именование идет через точку от сущности к конечному параметру. Пример: salary.from.label
3.  camelCase
4.  Все знаки должны быть в ключах, минимум конкатенации (лучше вообще никакой конкатенации переводов), тк есть справа-налево, есть языки где вначале предлложения ставятся какие-то знаки и еще много чего
