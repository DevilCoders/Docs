## Задача

Сейчас пользователи иногда пишут в комментарии водителю то, что относится к опциям 
(например, вводят информацию о необходимости детского кресла). 
Поэтому предлагается реализовать список требований рядом с экраном ввода комментария водителю.
В новом и в обычном списке могут содержаться одинаковые требования.

## Необходимые изменения

Для того, чтобы можно было показать требования на обоих экранах, 
предлагается вернуть два списка требований. 
Требования для экрана с комментарием могут содержать другой текст и картинку тарифа. 
Включать требования для экрана с комментарием будем под экспериментом.

## Решение
### Изменение в  `/zoneinfo`
Сейчас требования возвращаются на клиент в `supported_requirements`. 
Можно возвращать второй список рядом, для этого добавить в ответ `/zoneinfo` 
новое поле `supported_requirements_comment_screen`:
```
    ZoneinfoResponse:
        additionalProperties: false
        properties:
            ...
            max_tariffs:
                description: список тарифов
                items:
                    additionalProperties: false
                    properties:
                        ...
                        supported_requirements:
                            $ref: definitions.yaml#/definitions/supported_requirements
                            description: требования, которые можно указывать при заказе
                                в этом городе
                        supported_requirements_comment_screen:
                            $ref: definitions.yaml#/definitions/supported_requirements
                            description: требования, которые можно указывать при заказе
                                в этом городе, показываемые на экране ввода комментария для водителя
                        ...
```
Массив опций описан в `definitions/supported_requirements`, в новом поле можно вернуть опции того же формата.
Значение `label` нужно будет заменить на новое значение названия требования на экране, которое предлагается задавать в эксперименте.
В `definitions/supported_requirements` также можно добавить новое опциональное поле `trail_image` для изображения:
```
    supported_requirements:
        description: требования к заказу, регулируемые на стороне сервера
        items:
            additionalProperties: false
            properties:
                ...
                trail_image:
                    description: image_tag изображения, показываемое рядом с пожеланием 
                    type: string
            required:
              - name
              - label
              - type
              - persistent
            type: object
        type: array
```
Изображение может быть дополнительно загружено через админку для конкретного требования, 
если его нет - показывать изображение для соответствующего тарифа. 
Задать показ изображений можно в эксперименте.

Всё остальное флоу работы с требованиями остаётся то же, что и при выборе требований обычным образом.
Доработок на бэкенде для этого не требуется.

### Эксперимент

Предполагается, что эксперимент будет по пользователю, версии приложения и зоне, 
для разных зон можно будет сконфигурировать разное поведение. 
Эксперимент должен содержать список требований для показа на экране комментариев для разных тарифов,
и список требований для показа на экране тарифа для разных тарифов (списки могут не совпадать).
Текст для каждого требования и image tag для новой картинки.

Возможный вид эксперимента:
```
{
  "comment_screen_requirements": [
    {
      "name": "childchair",
      "label": "childchair_tanker_key",
      "image_tag": "childchair_image_tag",
      "tariff": [
        "business",
        "vip"
      ],
      "show_on_tariffs_screen": true
    }
  ]
}
```

Постепенно логику регулирования требований можно перенести в админку.
