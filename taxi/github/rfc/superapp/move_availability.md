# Проблема

- Сейчас, для того чтобы получить результирующую доступность сервисов цепочка запросов выглядит так:
`api-proxy/products` -> `superapp-misc/v1/services-positions` -> `api-proxy/products` -> `eda/v1/availability` -> `superapp-misc/v1/availability`, это негативно сказывается на таймингах, а также существует проблема с отсутсвием фолбеков на ручку определния позиции, из за сложной логики(в какой-то момент она была сложной для api-proxy).


# Решение

- Переносим запросы доступности еды в ручку `superapp-misc/availability`. Тем самым уменьшаем количество звеньев, что благоприятно должно сказаться на тайминги ручки `products` в целом и увеличивает её надёжность.
- Принимаем тот факт, что запросы на доступность внешних сервисов, будет осуществляться внутри ручки `superapp-misc/v1/availability`.

# Этапы:
1. Переносим запрос доступности еды в `superapp-misc`, делаем объект `services` опциональным, если данного объекта нет в запросе, необходимо сделать запрос в едовую ручку;
2. По эксперименту в `products`, отключаем хождение в ручку доступности еды и ручку выбора позиций. А также, в этом случае, не отправлять объект `services` в запрос ручки `superapp-misc/v1/availability`;
3. Перевести всех пользователей на новую схему работы;
4. Выпилить лишние запросы из `products` и выпилить ручку определений позиций из `superapp-misc`;
