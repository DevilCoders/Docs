#### Клиентская логика

Совместно с a-tumanov и vadim-b пришли к следующим принципам:
1) кэш глобальный на все приложение, где ключ - название эксперимента
2) в эндпойнт отправляются только ключи, которыми этот эндпойнт оперировал
3) у всех экспериментов в кэше есть внутреннее время жизни, которое выставляется клиентом (например 1 месяц), на тот случай, если от эндпойнта отказались (например переход с 3.0 на новую 4.0 ручку)
4) есть проблемы с георучками из-за комплексной логики использования экспериментов на стороне клиентов (у клиентов разная логика использования экспериментов для точек А и Б) - на данный момент считаем георучки некэшируемыми и если и будем разбираться, то отдельно
5) во всех разобранных кейсах гонок самое критичное, что может произойти - устаревание кэша, которое решается следующим запросом
* клиент при параллельных запросах должен сохранять локальные копии значения эксперимента до конца ответа ручки
* после ответа ручки - безусловно записывать в кэш значение (на стороне клиентов мы не храним логику сравнения версий)

#### Важные кейсы
* Отсутствие эксперимента в ответе == непопадание в эксперимент (даже если он был у клиента в кэше)
