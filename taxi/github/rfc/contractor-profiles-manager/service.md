**Название сервиса**
contractor-profiles-manager

**Какую продуктовую проблему решает сервис?**

Создание и редактирование исполнителей.

**Почему нельзя решить эту задачу без разработки нового сервиса существующими решениями?**

Сейчас нужная логика размазана на несколько сервисов:
1) parks, сервис монолит в deprecated репозитории. Основная логика находится здесь. 
2) taxi-fleet, сервис монолит. Сервис проксирует ручку из parks и добавляет немного своей логики.

С текущей архитектурой:
1) Логика дублируется между сервисами.
2) Повышенная сложность поддержки обратной совместимости.
3) Увеличенные тайминги из за лишнего проксирования.

**Как именно сервис будет решать поставленные перед ним задачи?**

Будет 2 этапа:
1) Новый сервис заберет себе всю логику с taxi-fleet, но также будет ходить в parks. Заодно эта логика будет оптимизирована для фронта superweb.
2) Перетаскиваем в сервис логику parks.

В итоге будет цепочка
superweb -> fleet-authproxy -> contractor-profiles-manager -> база данных

**Разрабатывается один сервис или система?**

Один сервис.

**С кем взаимодействует сервис?**

fleet-authproxy
parks

**Какие базы использует?**

На первом этапе не требуется.
На втором переиспользуем dbdrivers.drivers, возможно taximeter_changes.

Нет

Да

Данные исполнителей

Зависит от rps, там совсем немного
