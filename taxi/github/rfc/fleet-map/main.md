### Архитектурное описание сервиса - fleet-maps

### Название сервиса
fleet-maps

### Какую продуктовую проблему решает сервис?
Новая итерация карты с машинками/водителями для диспетчерской в Superweb, с переиспользованием 
в международке Saas, заменит карту из Оптеум (Диспа 2.0) и карту из старой шарповой Диспы.

### Почему нельзя решить эту задачу без разработки нового сервиса существующими решениями?
Сервис выглядит нагруженным (зависит от потенциала международки Saas), поэтому выбираем плюсы
и выделяем как отдельный микросервис и продуктово и технически. Старая ручка для карты Оптеум 
(Диспа 2.0) более не используется находиться в сервисе taxi-fleet на python. Сервис taxi-fleet 
является монолитом от которого отказываемся в пользу микросервисной архитектуры и не селим 
туда более ничего)

### Как именно сервис будет решать поставленные перед ним задачи?
Сервис будет получать данные из источников и отдавать данные для построения карты, детализацию 
по сущностям (водители, заказы, авто?) и справочники для диспетчерской

### Разрабатывается один сервис или система?
Один сервис

### Какая нагрузка ожидается?
1. Ожидается от 25 RPS, взято за последние 6 месяцев с ручки обновления карты из Оптеум (Диспа 2.0)
2. Сейчас примерно 200-300 RPS на шарповую диспу
3. И дополнительно рост от международки Saas (100 RPS?)ß
~ 350 RPS

### Какие фолбеки предусмотрены на сам этот сервис?
Частичная деградация отдельный компонентов, задержки по перерисовке водителей/авто/заказов

### Какие фолбеки предусмотрены внутри этого сервиса на взаимодействие с другими сервисами?
Конфиги с настройками таймаутов

### Какие возможности масштабируемости закладываются?
Горизонтальное и вертикальное масштабирование

### Какие точки отказа есть в сервисе?
Недоступность отдельных сервисов из которых будем получать данные
