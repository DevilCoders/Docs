## Бизнесовая задача

Сервис отвечат за букинг водителя.

## Архитектура

### API

Ручка `GET /v1/status_info`:

```json
{
    "order_id": "some_order_id"
}
```

```json
{
  "status_info": {},
  "max_waiting_time": "",
  "departure_time": ""
}
```

### v1/status_info
Проверяет, если букинг, то заполняет status_info.
Предполагается, что или букинг или предзаказ могут заполнить status_info. Насколько я понимаю, вместе они произойти не могут.
Если могут, нужно делать это в одном сервисе.
####  Fallback
Можно не показывать. 

### Технические детали и ограничения
1. Набор заполняемых для totw полей совпадает с фичей про предзаказ. Предполагается, что случится или одно или другое и соответствующий сервис заполнит эти поля. Но если они могут произойти вместе, то нужно разобраться, кто приоритетнее и оставить только одну ручку.
2. Сервис ради одной ручки тоже неидеальное решение. Нужно подумать, может ли данный сервис забрать еще какую-то функциональность.
