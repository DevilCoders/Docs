# Архивирование пользователя

Запрос позволяет архивировать пользователя клиента. При архивации пользователь пропадает из личного кабинета. Пользователь не может пользоваться сервисами, и его нельзя найти в личном кабинете.

При попытке создать пользователя с таким же номером телефона, как и у архивированного пользователя, возникнет ошибка. В этом случае вместо создания нового пользователя следует [восстановить пользователя из архива](#restore) с обновленными данными.

## Синтаксис запроса

```
GET https://b2b-api.go.yandex.ru/integration/2.0/users/{идентификатор пользователя}/archive
```

{% include [user-create-headers-title](../../_includes/concepts/api20/user-create/id-user-create/headers-title.md) %}


#### Authorization
OAuth-токен. Процесс получения токена описан в разделе [Начало работы](../quickstart.md).

## Описание полей ответа

В ответе содержится поле:

Поле | Описание | Формат
----- | ----- | -----
`id` | Идентификационный номер пользователя. | Строка


## Пример запроса

```
GET https://b2b-api.go.yandex.ru/integration/2.0/users/f65...c57d/archive
...
Authorization: Bearer <OAuth-токен>
```

## Пример ответа

Пример ответа на данный запрос выглядит следующим образом:

```no-highlight
{
        "id": "3caa3587675b49deb62e3286b753b05e"
}
```

## Возможные коды ответа

Ответ на данный запрос может содержать следующие стандартные HTTP-коды:

- `200` — запрос выполнен успешно.
- `401` — был передан неверный [OAuth-токен](../quickstart.md).
- `403` — у клиента не хватает прав на выполнение данного запроса.
- `404` — запрашиваемая запись не найдена.


## Восстановление архивированного пользователя {#restore}

Для восстановления нужно отправить запрос на [редактирование пользователя](user-update.md) с добавлением флага `is_deleted: false`.
