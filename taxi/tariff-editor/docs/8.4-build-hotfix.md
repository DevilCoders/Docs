Фикс надо сделать так чтобы он попал сначала в релизную ветку, затем в `master`. Поскольку релизная ветка отведена от ветки `stable`, в которой коммиты с версиями, которых не должно быть в `master`, от неё бранчеваться для фикса нельзя. Таким образом надо делать бранч от нужного коммита в `master`.

Допустим наш релиз был версии `0.0.1460.1`, тогда порядок действий следующий:
1. Поскольку версия релиза получается добавлением четвертого числа к RC версии, исходная версия была `rc-0.0.1460`. Находим коммит с этой версией в `stable`

![image](https://media.github.yandex-team.ru/user/5576/files/1c210e00-ce73-11e9-8e01-9f6ef4d97198)

2. Берем первый коммит из `master`, который идет перед коммитом с версией, обычно это коммит с публикацией пакетов, и копируем его хэш

![image](https://media.github.yandex-team.ru/user/5576/files/728e4c80-ce73-11e9-8d30-792d4e7ccfad)

3. Откатываем у себя `master` до этого коммита выполнив `git reset --hard <hash>`. Проверяем через `git log`.

![image](https://media.github.yandex-team.ru/user/5576/files/ef212b00-ce73-11e9-8ac2-8633299573d6)

4. Делаем бранч от этого состояния истории `git checkout -b release-0.0.1460-fix`
5. Делаем фикс в этой ветке

![image](https://media.github.yandex-team.ru/user/5576/files/61463f80-ce75-11e9-8d09-8b6aa40f5fcc)

6. Делаем PR в релизную ветку. В `unstable` это катать не надо, фиксы проверяются разработчиком, затем уходят в `testing` и там проверяются QA.

![image](https://media.github.yandex-team.ru/user/5576/files/0d882600-ce76-11e9-943e-61f8d14dffc6)

7. Мерджим PR в релизную ветку, но НЕ УДАЛЯЕМ ветку с фиксом. Тут ревью не требуется.
8. Запускаем еще раз `tariff-editor-make-release` на нужную версию `rc-<version>`.
9. Процесс фиксов может повторятся до тех пор пока QA не скажут, что с релизом все ок, все фиксы делаются все в той же ветке, коммиты не схлопываем.

Допустим у нас было два фикса\
Таким образом мы получаем в ветке для фиксов `release-0.0.1460-fix`

![image](https://media.github.yandex-team.ru/user/5576/files/83df5500-ce83-11e9-9630-214af9ed6b71)

А в релизной ветке `release-0.0.1460`

![image](https://media.github.yandex-team.ru/user/5576/files/b721e400-ce83-11e9-83ee-4013500fe256)

10. Когда QA говорят, что с релизом все ок, катим итоговый кондукторный тикет из тестинга в престейбл (в нашем примере - это тикет `0.0.1460.3`), далее как обычно собираем апрувы и катим в стейбл.

11. Затем нужно замерждить фиксы в `master`. Для этого необходимо завести еще одну ветку от ветки с фиксами, поскольку сама ветка с фиксами может еще понадобится, если какие-то баги все же пропустили и они обнаружатся до следующего релиза. Назовем эту ветку `release-0.0.1460-fix-to-master`. Делаем ребейз этой ветки с `master`.

```bash
git checkout master
git pull
git checkout release-0.0.1460-fix
git pull
git checkout -b release-0.0.1460-fix-to-master
git rebase master # решаем конфликты
git push origin release-0.0.1460-fix-to-master
```
В итоге видим в нашей ветке

![image](https://media.github.yandex-team.ru/user/5576/files/f8b48e00-ce87-11e9-9e3c-c460e1c536c0)

12. Делаем PR `release-0.0.1460-fix-to-master` в `master`.

![image](https://media.github.yandex-team.ru/user/5576/files/7b3d4d80-ce88-11e9-8439-02198390923f)

Поскольку ветка на актуальном мастере, можно катнуть ее в `unstable`, отдать посмочить, убедится что наши фиксы нормально легли на мастер. 

13. Проходим ревью и мерджим этот PR в `master`. Удаляем ветку `release-0.0.1460-fix-to-master`.
