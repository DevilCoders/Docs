В какой-то момент времени менеджер выбирает, какая версия из release candidate пойдет в релиз

Для того чтобы собрать эту версию и катнуть в `testing` надо запустить сборку в проекте `tariff-editor-make-release`.

Допустим менеджер попросил собрать версию на базе `rc-0.0.1452`

![image](https://media.github.yandex-team.ru/user/5576/files/6631c300-cd91-11e9-950b-b26bb68854ac)

Далее откроется окно запуска билда, выбираем там следующее:
![image](https://media.github.yandex-team.ru/user/5576/files/99549580-4db8-11ea-9c2b-f27268842b00)

Во время сборки происходи следующее:
- Происходит checkout выбранного тега (Teamcity скрывает префикс `rc-`)
- Из тега происходит checkout ветки `release-<version>`, если ветки нет, то она создается, если есть, то просто происходит переключение и git pull ветки, на случай если в нее были замерджены хотфиксы
- Происходит ап версии релиза, которая добавляется четвертым числом к версии RC, те если был тег `rc-0.0.1452`, то первый релиз будет версии `0.0.1452.1`, затем `0.0.1452.2` и тд.
- Собирается changelog релиза (если не было хотфиксов с момента создания ветки, или прошлого релиза из этой ветки, то он будет содержать просто новую версию, дату и вот это все).
- Прогоняются тесты, происходит сборка и аплоад статики в s3 и debian пакета в репозиторий verstka
- Создается кондукторный тикет на раскатку debian пакета на машинки в `testing`
- Происходит коммит с ченджлогом и пуш ветки в репозиторий
